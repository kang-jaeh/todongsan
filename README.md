# 동네대전 🏘️

> 지역 선택에 필요한 집단지성을 수집하는 플랫폼.  
> 주관적 선호는 **Battle 블라인드 투표**로 수집하고, 객관적으로 판정 가능한 지역 지표는 **Market 포인트 예측 시장**으로 운영하여 사용자가 더 합리적인 지역 선택을 할 수 있도록 돕는다.

SSAFY 1학기 관통프로젝트 | Java Track | Back-End (Spring Boot)

---

## 1. 서비스 개요

사람들이 매일 맞닥뜨리는 지역 선택 고민에는 정답이 없다.

```text
어디로 이사 가지?
주말에 어디 여행 가지?
데이트 어디서 하지?
어디서 창업하지?
어느 동네가 앞으로 더 뜰까?
```

하지만 다른 사람들의 선택과 시장의 흐름은 존재한다.

**동네대전**은 지역에 대한 사람들의 선호, 경험, 예측을 수집하고 이를 데이터로 시각화하여 사용자가 더 합리적인 지역 선택을 할 수 있도록 돕는 서비스다.

단순한 부동산 정보 조회 서비스가 아니라,  
**지역 선택에 대한 집단지성 플랫폼**을 지향한다.

---

## 2. 핵심 문제의식

기존 부동산/지역 정보 서비스는 보통 정량 데이터 중심이다.

- 매매가
- 전세가
- 월세
- 평수
- 역세권 여부
- 주변 편의시설

하지만 실제 사용자는 이런 질문을 더 많이 한다.

```text
실제로 살기 좋은 곳은 어디인가?
데이트하기 좋은 곳은 어디인가?
자취 초보에게 더 나은 동네는 어디인가?
앞으로 가격이 오를 가능성이 높은 지역은 어디인가?
사람들은 지금 어떤 지역을 더 선호하고 있는가?
```

동네대전은 이 질문들을 두 가지 방식으로 해결한다.

1. **Battle**: 사람들의 주관적 선호를 블라인드 투표로 수집한다.
2. **Market**: 객관적으로 판정 가능한 지역 지표를 포인트 예측 시장으로 운영한다.

---

## 3. 사용자 흐름

```text
1. 사용자는 Battle에서 지역 선택 투표에 참여한다.
   예: "성수 vs 연남, 데이트하기 어디가 더 좋을까?"

2. 투표 참여 보상으로 서비스 내부 Point를 획득한다.

3. 투표 종료 후 결과를 확인한다.
   - 전체 결과
   - 연령대별 결과
   - 성별별 결과
   - 거주지역별 결과
   - 방문 인증 사용자 결과

4. 사용자는 Battle에서 획득한 Point로 Market 예측에 참여할 수 있다.
   예: "다음 주 서울 아파트 매매가격지수는 상승할까?"

5. Market은 공공데이터 또는 외부 지표로 결과를 확정한다.

6. 정답 선택지 참여자들은 전체 참여 포인트 풀에서 수수료를 제외한 정산 대상 풀을 참여 비율대로 나눠 가진다.

7. 누적된 투표, 예측, 댓글, 인증 데이터는 Insight에서 분석된다.

8. 사용자는 AI 요약과 통계를 통해 더 합리적인 지역 선택을 할 수 있다.
```

## 4. 핵심 설계 원칙

### 4-1. Battle과 Market은 분리한다

> **참여자가 결과를 직접 바꿀 수 있는 주제에는 포인트 예측 참여를 허용하지 않는다.**

아래와 같은 주제는 사용자의 투표 결과 자체가 승패를 결정한다.

```text
강남 vs 강북, 데이트하기 어디가 더 좋은가?
성수 vs 연남, 주말에 놀기 어디가 더 좋은가?
신림 vs 건대, 자취하기 어디가 더 좋은가?
```

이런 주제에 포인트 예측을 허용하면 사용자의 선호가 왜곡될 수 있다.

사용자가 실제로 선호하는 곳이 아니라,  
"이길 것 같은 곳"에 투표할 가능성이 생기기 때문이다.

따라서 이런 주제는 **Battle**에서만 운영하며, 포인트 예측 참여는 허용하지 않는다.

---

### 4-2. Market은 객관적으로 판정 가능한 주제만 허용한다

Market은 사용자의 투표가 아니라 외부 데이터로 결과가 확정되는 주제만 다룬다.

좋은 Market 주제의 조건은 다음과 같다.

- 결과를 객관적으로 판정할 수 있어야 한다.
- 판정 데이터 출처가 명확해야 한다.
- 마감일과 정산일이 명확해야 한다.
- 사용자가 직접 결과를 조작하기 어려워야 한다.
- YES/NO 또는 선택지가 명확해야 한다.
- 지역 비하, 혐오, 정치적 선동으로 흐르지 않아야 한다.

---

### 4-3. Point는 현금성 자산이 아니다

Point는 서비스 참여를 유도하는 내부 재화다.

- 현금 환전 불가
- 코인 전환 불가
- 상품권/기프티콘 교환 불가
- 사용자 간 양도 불가
- 외부 거래 불가

Point는 오직 서비스 내부에서만 사용된다.

---

### 4-4. Point 경제 구조

동네대전의 Point 경제는 다음 흐름을 가진다.

```text
Battle = Point 생성
Market = Point 사용 / 이동 / 일부 소각
Insight = Point 소비
Profile = Point 소비
```

Battle은 사용자 참여를 유도하기 위해 Point를 발행한다.  
Market은 사용자가 보유 Point로 예측에 참여하고, 정답자가 전체 풀을 나눠 갖는 구조다.  
이때 전체 참여 포인트의 일부를 시스템 수수료로 소각하여 포인트 인플레이션을 완화한다.

## 5. 최종 기술 방향

### 5-1. REST 기반 MSA

MVP는 **REST 기반 MSA**로 구현한다.

다만 Kafka, RabbitMQ 등 이벤트 브로커 기반 비동기 처리는 MVP에서 제외한다.

```text
Client
  └── API Gateway
        ├── Member-Point Service
        ├── Battle Service
        ├── Market Service
        └── Insight-Reputation Service
```

### 5-2. 서비스 책임

| 서비스 | 책임 |
|---|---|
| API Gateway | 외부 요청 라우팅, JWT 검증, 인증 정보 전달 |
| Member-Point Service | 회원, OAuth/JWT, Point 지갑, Point 히스토리, Point 적립/차감/정산 |
| Battle Service | Battle 주제 등록/검수, 블라인드 투표, 댓글, 결과 통계 |
| Market Service | Market 주제 등록/검수, 예측 참여, 예상 정산 비율 계산, 정산/무효 처리 |
| Insight-Reputation Service | 사용자 신뢰도, 방문 인증, AI 요약, 데이터 분석 |

### 5-3. REST 연계 원칙

- 서비스 간 통신은 REST API 직접 호출로 처리한다.
- 이벤트 브로커는 사용하지 않는다.
- 모든 요청/응답은 공통 응답 포맷을 따른다.
- Point 관련 API는 멱등성 키를 사용하여 중복 지급/중복 차감을 방지한다.
- Market 예측 참여는 Point 차감 성공 후에만 확정 저장한다.

### 6-1. 주요 호출 흐름

| 호출 방향 | 트리거 시점 | 목적 |
|---|---|---|
| Gateway → Member-Point | 인증 요청 | JWT 검증 및 사용자 정보 조회 |
| Battle → Member-Point | 투표 완료 | 투표 참여 Point 지급 |
| Battle → Member-Point | 댓글 작성 완료 | 댓글 작성 Point 지급 |
| Battle → Member-Point | Battle 주제 승인 | 주제 등록 승인 Point 지급 |
| Battle → Insight-Reputation | 투표/댓글 완료 | 활동 점수 업데이트 |
| Market → Member-Point | 예측 참여 요청 | 참여 Point 차감 |
| Market → Member-Point | 정산 완료 | 정산 보상 Point 지급 |
| Market → Member-Point | Market 무효 처리 | 참여 Point 환불 |

| Insight-Reputation → Battle | AI 분석 요청 | Battle 투표 원본 데이터 조회 |

| Insight-Reputation → 공공 API | AI 분석 요청 | 부동산/거래량 등 지역 지표 조회 |

---

### 6-2. Battle 투표 완료 → Point 지급

Battle은 사용자 투표를 먼저 저장하고, 이후 Member-Point Service에 투표 참여 보상을 요청한다.

Point 지급 실패가 발생해도 투표 자체는 유지된다.  
이 경우 Battle Service는 Point 지급 재시도 큐에 실패 건을 적재하고 재시도한다.

```text
Client → Gateway → Battle        POST /battles/{battleId}/votes
Battle → DB                      투표 저장
Battle → Member-Point            POST /points/earn
Member-Point → Battle            200 OK
Battle → Client                  투표 완료 응답
```

Point 지급 실패 시:

```text
Battle → DB                      point_reward_retry_queue 적재
Battle → Client                  "투표 완료. 포인트는 잠시 후 지급됩니다."
Scheduler → Member-Point         1분 간격 재시도, 최대 3회
3회 실패 시                       관리자 수동 보정 대상으로 기록
```

---

### 6-3. Market 예측 참여 → Point 차감 → 참여 확정

Market 예측 참여는 Point 차감 성공이 선행되어야 한다.

```text
Client → Gateway → Market        POST /markets/{marketId}/predictions
Market → DB                      PENDING 예측 참여 기록 생성
Market → Member-Point            POST /points/spend
Member-Point → Market            차감 성공
Market → DB                      예측 참여 기록 CONFIRMED 전환
Market → Client                  예측 참여 완료 응답
```

Point 차감 실패 시:

```text
Market → DB                      예측 참여 기록 FAILED 처리
Market → Client                  예측 참여 실패 응답
```

예측 참여가 실패하는 대표 상황은 다음과 같다.

- 보유 Point 부족
- Market 마감
- 동일 Market 중복 참여
- 유효하지 않은 선택지
- Point Service 장애 또는 타임아웃

> **정산 대상에는 CONFIRMED 상태의 예측 참여 기록만 포함한다.**

---

### 6-4. Market 정산 → Point 지급

Market 정산은 관리자가 결과를 입력하여 실행한다.

```text
관리자 → Market                  POST /markets/{marketId}/settle
Market → DB                      CONFIRMED 참여 기록 조회
Market → DB                      정산 금액 계산
Market → Member-Point            POST /points/settlements
Member-Point → DB                사용자별 정산 보상 지급
Market → DB                      Market 상태 SETTLED 전환
Market → 관리자                  정산 완료 응답
```

정산 지급 API는 `settlementId`를 멱등성 키로 사용하여 중복 정산을 방지한다.

```json
{
  "success": true,
  "errorCode": null,
  "message": null,
  "data": {},
  "timestamp": "2026-05-28T10:00:00"

```

### 7-2. 에러 응답

```json
{
  "success": false,
  "errorCode": "POINT_INSUFFICIENT",
  "message": "포인트가 부족합니다. 현재 보유: 50P, 필요: 100P",
  "data": null,
  "timestamp": "2026-05-28T10:00:00"
}
```

### 7-3. 주요 에러 코드

| 에러 코드 | HTTP 상태 | 설명 |
|---|---:|---|
| UNAUTHORIZED | 401 | JWT 토큰 없음 또는 만료 |
| FORBIDDEN | 403 | 권한 없음 |
| NOT_FOUND | 404 | 리소스 없음 |
| POINT_INSUFFICIENT | 400 | Point 부족 |
| ALREADY_VOTED | 409 | 이미 투표한 Battle |
| ALREADY_PREDICTED | 409 | 이미 참여한 Market |
| MARKET_CLOSED | 400 | 마감된 Market |
| BATTLE_CLOSED | 400 | 종료된 Battle |
| INVALID_SETTLE | 400 | 정산 조건 미충족 |
| INTERNAL_ERROR | 500 | 서버 내부 오류 |

## 8. Battle — 선호 투표

Battle은 지역에 대한 사람들의 주관적 선호를 수집하는 기능이다.

### 8-1. 예시 주제

```text
성수 vs 연남, 데이트하기 어디가 더 좋을까?
신림 vs 건대, 자취하기 어디가 더 좋을까?
강릉 vs 여수, 여름 여행 어디가 더 좋을까?
판교 vs 광교, 직장인이 살기 어디가 더 좋을까?
홍대 vs 을지로, 맛집 탐방하기 어디가 더 좋을까?
```

### 8-2. 주요 기능

- 사용자가 직접 Battle 주제 등록
- 관리자가 Battle 주제 등록
- 사용자 등록 주제 검수
- 블라인드 투표
- 투표 전 결과 비공개
- 투표 후 결과 공개
- 댓글/토론 기능
- AI 투표 결과 요약

---

### 8-3. 블라인드 투표 정책

투표 전에는 현재 투표 결과를 보여주지 않는다.

이유는 다음과 같다.

- 밴드왜건 효과 방지
- 초기 투표 쏠림 방지
- 사용자의 순수 선호 수집
- 데이터 왜곡 방지

투표를 완료한 사용자만 현재 결과를 볼 수 있다.

---

### 8-4. Battle 결과 공개 정책

| 케이스 | 공개 여부 | 공개 범위 |
|---|---|---|
| 투표 진행 중 + 미투표 회원 | 비공개 | 현재 참여자 수만 표시 |
| 투표 진행 중 + 투표 완료 회원 | 부분 공개 | 전체 득표 비율 + 현재 참여자 수 |
| 투표 진행 중 + 비회원 | 비공개 | 현재 참여자 수만 표시 |
| 투표 종료 직후 + 투표 완료 회원 | 공개 | 전체 결과 + 기본 교차분석 |
| 투표 종료 직후 + 미투표 회원 | 비공개 | 없음 |
| 투표 종료 직후 + 비회원 | 비공개 | 없음 |
| 투표 종료 후 72시간 경과 + 미투표 회원 | 공개 | 전체 결과 비율만 |
| 투표 종료 후 72시간 경과 + 비회원 | 공개 | 전체 결과 비율만 |

---

### 8-5. 공개 범위 단계

| 공개 단계 | 포함 데이터 | 접근 조건 | 비용 |
|---|---|---|---:|
| 기본 공개 | 전체 득표 비율 + 참여자 수 | 투표 완료 회원 | 무료 |
| 결과 공개 | 전체 득표 비율 + 연령대별 + 성별별 결과 | 종료 후 투표 완료 회원 | 무료 |
| 상세 교차분석 | 거주지역별 상세 교차분석 | Point 소비 | 30P |
| 방문 인증자 필터 | 방문 인증 사용자만 필터링한 결과 | Point 소비 | 30P |
| AI 리포트 | AI 자연어 분석 요약 | Point 소비 | 80P |

### 8-6. Battle 엣지케이스

| 상황 | 처리 |
|---|---|
| 투표 동률 | 동률 표시, 승리 보너스 미지급 |
| 투표 참여자 10명 미만 | 통계적 의미 부족으로 결과 비공개, 참여 보상만 지급 |
| Battle 강제 종료 | 결과 비공개, 참여 보상 유지, 승리 보너스 미지급 |
| 관리자 조기 종료 | 조기 종료 안내 후 종료 기준에 따라 처리 |

---

## 9. Market — 포인트 예측 시장

Market은 객관적 데이터로 판정 가능한 지역 관련 이벤트를 예측하는 기능이다.

사용자는 Battle에서 획득한 Point를 사용해 Market 예측에 참여할 수 있다.

---

### 9-1. Market 예시

#### 주간 가격지수 예측

```text
다음 주 서울 아파트 매매가격지수는 상승할까?
다음 주 서울 아파트 전세가격지수는 상승할까?
다음 주 강남구 아파트 매매가격지수 변동률은 0.05% 이상일까?
다음 주 수도권 아파트 전세가격지수는 상승할까?
```

#### 월간 거래량 예측

```text
이번 달 서울 아파트 거래량은 지난달보다 증가할까?
이번 달 마포구 주택 거래량은 전월 대비 10% 이상 증가할까?
이번 달 성동구 아파트 거래량은 전월보다 증가할까?
```

#### 월간 가격동향 예측

```text
이번 달 서울 주택 매매가격지수는 전월 대비 상승할까?
이번 달 성동구 아파트 전세가격지수는 상승할까?
이번 달 오피스텔 월세가격지수는 상승할까?
```

#### 실거래가 스페셜 Market

```text
2026년 6월 은마아파트 전용 84㎡ 최고 실거래가는 30억 이상일까?
2026년 6월 헬리오시티 전용 84㎡ 최저 실거래가는 20억 이하일까?
```

실거래가 기반 Market은 데이터 공개 지연과 거래 없음 이슈가 있으므로 MVP에서는 스페셜 Market으로 제한한다.

#### 정책/개발 이벤트 예측

```text
올해 안에 특정 GTX 노선 착공 발표가 날까?
올해 안에 특정 재개발 구역 사업시행인가가 승인될까?
올해 안에 특정 지역 토지거래허가구역이 해제될까?
```

정책/개발 이벤트는 판정 기준이 모호해질 수 있으므로 관리자 등록 주제로만 운영한다.

---

### 9-2. Market 판정 필수 정보

Market은 반드시 등록 시점에 판정 규칙을 함께 명시해야 한다.

| 항목 | 설명 |
|---|---|
| 제목 | 사용자가 이해하기 쉬운 예측 질문 |
| 선택지 | YES/NO 또는 명확한 다중 선택지 |
| 판정 데이터 | 결과 확정에 사용할 데이터 출처 |
| 기준일 | 어떤 날짜의 데이터를 기준으로 판단할지 |
| 마감일 | 예측 참여가 종료되는 시점 |
| 정산일 | 결과를 확정하고 Point를 지급하는 시점 |
| 무효 조건 | 데이터 미공개, 거래 없음 등 예외 상황 |
| 관리자 검수 여부 | Market 등록 전 검수 필수 |

---

### 9-3. Market 판정 예시 — 주간 가격지수

```text
주제:
다음 주 서울 아파트 매매가격지수는 상승할까?

선택지:
YES / NO

판정 데이터:
한국부동산원 주간 아파트 가격동향

판정 기준:
발표된 서울 아파트 매매가격지수 변동률이 0보다 크면 YES,
0 이하이면 NO로 판정한다.

마감일:
해당 주 발표 전날 23:59

정산일:
공식 데이터 발표 후 관리자 확인 또는 자동 수집 후 정산
```

---

### 9-4. Market 판정 예시 — 월간 거래량

```text
주제:
2026년 6월 서울 아파트 거래량은 2026년 5월보다 증가할까?

선택지:
YES / NO

판정 데이터:
한국부동산원 또는 국토교통부 거래량 통계

판정 기준:
2026년 6월 서울 아파트 거래량이 2026년 5월 거래량보다 많으면 YES,
같거나 적으면 NO로 판정한다.

정산일:
공식 월간 거래량 데이터 공개 후 정산

예외:
데이터 공개가 지연될 경우 최대 7일 대기한다.
이후에도 공개되지 않으면 관리자가 Market을 무효 처리할 수 있다.
```

---

### 9-5. Market 판정 예시 — 실거래가 스페셜 Market

```text
주제:
2026년 6월 은마아파트 전용 84㎡ 최고 실거래가는 30억 이상일까?

선택지:
YES / NO

판정 데이터:
국토교통부 실거래가 공개시스템

판정 기준:
계약일이 2026년 6월인 은마아파트 전용 84㎡ 거래 중
공개된 최고 실거래가가 30억 이상이면 YES,
30억 미만이면 NO로 판정한다.

거래 없음:
해당 기간 거래가 없을 경우 Market은 무효 처리한다.

해제 거래:
해제 신고된 거래는 정산 대상에서 제외한다.

정산일:
2026년 7월 말 기준 공개 데이터로 정산한다.

주의:
정산 이후 추가 신고 또는 정정 데이터가 올라오더라도 결과는 변경하지 않는다.
```

## 10. Point — 참여 보상 경제

Point는 서비스 내부에서 사용하는 참여 보상 재화다.

Point의 목적은 다음과 같다.

- Battle 참여 유도
- Market 참여 유도
- 커뮤니티 활동 유도
- 사용자 리텐션 강화
- 서비스 내부 게임화
- 지역 선택 데이터 축적

---

### 10-1. Point 획득 방법

| 활동 | 지급 Point | 제한 | 비고 |
|---|---:|---|---|
| 신규 가입 | 50P | 계정당 1회 | 첫 Market 체험 유도 |
| Battle 투표 참여 | 10P | 동일 Battle 1회 | 기본 참여 보상 |
| Battle 승리 진영 선택 | +2P | 동일 Battle 1회 | 투표 종료 후 추가 지급 |
| Battle 댓글 작성 | 2P | 동일 Battle 1회, 일일 최대 3건 | 도배 방지 |
| Battle 주제 등록 승인 | 20P | 승인 시 1회 | 검수 미통과 시 미지급 |

> Market 예측 참여 자체에는 Point를 지급하지 않는다.  
> Point는 Battle에서 벌고, Market에서 사용하는 구조를 유지한다.

---

### 10-2. Point 소비 방법

| 소비처 | 소요 Point | 소비 유형 | 비고 |
|---|---:|---|---|
| Market 예측 참여 | 최소 10P / 최대 500P | 정산 시 반환 가능 | 동일 Market 중복 참여 불가 |
| 거주지역별 상세 교차분석 열람 | 30P | 소비 | 해당 Battle 건당 |
| 방문 인증자 필터링 결과 열람 | 30P | 소비 | 해당 Battle 건당 |
| AI 상세 분석 리포트 생성 | 80P | 소비 | 건당 |
| Battle 주제 생성권 | 30P | 소비 | 검수 미통과 시 환불 없음 |
| 관심 지역 슬롯 확장 | 50P/슬롯 | 소비 | 기본 3개 무료, 최대 7개 |

---

### 10-3. 일일 획득 제한

포인트 파밍과 어뷰징을 방지하기 위해 Battle 기반 일일 획득 상한을 둔다.

```text
Battle 기반 일일 최대 획득 Point: 100P
```

단, Market 정산 보상은 일일 획득 제한에 포함하지 않는다.

---

## 11. Market 정산 방식 — 풀 배분 + 수수료 소각

Market 정산은 **풀 배분 방식**으로 처리한다.

많이 몰린 선택지는 예상 정산 비율이 낮아지고, 적게 몰린 선택지는 예상 정산 비율이 높아진다.

---

### 11-1. 핵심 공식

```text
전체 참여 포인트 풀 = 모든 CONFIRMED 예측 참여 포인트 합계

시스템 수수료 = 전체 참여 포인트 풀 × 5%

정산 대상 풀 = 전체 참여 포인트 풀 - 시스템 수수료

예상 정산 비율 = 정산 대상 풀 / 해당 선택지 참여 포인트 풀

개인 정산액 = 본인 참여 포인트 × 최종 정산 비율
```

---

### 11-2. 수수료 소각 정책

Market 정산 시 전체 참여 포인트 풀의 **5%**를 시스템 수수료로 소각한다.

수수료 소각 목적은 다음과 같다.

- Battle에서 생성되는 Point 인플레이션 완화
- Point 희소성 유지
- Market 참여 긴장감 유지
- 서비스 내부 포인트 경제 안정화

수수료는 운영자 수익으로 지급되지 않고 시스템에서 제거된다.

---

### 11-3. 정산 계산 정밀도

Point는 소수점 둘째 자리까지 저장한다.

정산 계산 결과는 소수점 둘째 자리까지만 인정하며,  
**소수점 셋째 자리 이하는 버림 처리**한다.

버림 처리로 인해 남는 잔여 Point는 시스템 소각 처리한다.

```text
예:
계산 결과: 33.337P
최종 지급: 33.33P
잔여 0.007P: 소각
```

---

### 11-4. 정산 예시 A — 소수 선택지 승리

| 항목 | 수치 |
|---|---:|
| YES 참여 총합 | 100P |
| NO 참여 총합 | 300P |
| 전체 풀 | 400P |
| 수수료 5% | 20P |
| 정산 대상 풀 | 380P |
| 결과 | YES 승리 |
| YES 최종 정산 비율 | 380 / 100 = 3.8배 |
| YES에 50P 참여한 사용자 정산액 | 50 × 3.8 = 190P |
| 순이익 | 190P - 50P = +140P |

---

### 11-5. 정산 예시 B — 다수 선택지 승리

| 항목 | 수치 |
|---|---:|
| YES 참여 총합 | 300P |
| NO 참여 총합 | 100P |
| 전체 풀 | 400P |
| 수수료 5% | 20P |
| 정산 대상 풀 | 380P |
| 결과 | YES 승리 |
| YES 최종 정산 비율 | 380 / 300 = 1.266...배 |
| YES에 150P 참여한 사용자 정산액 | 150 × 1.266... = 189.99P |
| 순이익 | 189.99P - 150P = +39.99P |

---

### 11-6. 승리 선택지 참여자가 없는 경우

예측 결과는 확정되었지만 해당 선택지에 참여한 사용자가 없는 경우 정산 대상자가 존재하지 않는다.

```text
YES 참여 총합: 1000P
NO 참여 총합: 0P
결과: NO
```

이 경우 전체 참여 포인트 풀은 정산되지 않고 전액 소각된다.

---

### 11-7. 무효 처리 시 환불 정책

| 무효 사유 | 참여 포인트 처리 |
|---|---|
| 공공 API 데이터 미수신 7일 초과 | 전액 환불 |
| 관리자 오류 | 전액 환불 |
| Market 조기 취소 | 전액 환불 |
| 부분 데이터 누락으로 판정 불가 | 전액 환불 |
| 해당 기간 실거래 없음 | 전액 환불 또는 사전 정의된 무효 조건 적용 |

무효 처리 시 수수료 소각은 발생하지 않는다.

---

## 12. Reputation — 신뢰도 시스템

Reputation은 사용자의 활동 신뢰도를 나타내는 시스템이다.

동네대전은 단순히 많은 의견을 모으는 것이 아니라,  
**누가 말했는가**에 대한 신뢰도도 함께 제공한다.

---

### 12-1. Reputation 구성 요소

| 요소 | 설명 |
|---|---|
| 활동 점수 | 투표, 댓글, 주제 등록 등 활동 기반 |
| 예측 정확도 | Market 예측 성공률 기반 |
| 거주지역 선언 | 사용자가 직접 선택한 거주지역 |
| 방문 인증 | GPS 또는 댓글 기반 방문 경험 인증 |
| 신고 이력 | 부적절한 활동이 많을 경우 감점 |

---

### 12-2. 거주지역 인증

MVP에서는 문서 기반 실거주 인증을 제외한다.

대신 사용자가 거주 지역을 시/구 단위로 직접 선택한다.

| 항목 | 내용 |
|---|---|
| 인증 방법 | 사용자가 거주 지역을 시/구 단위로 직접 선택 |
| 신뢰도 보정 | 해당 지역 Battle에서 투표 또는 댓글 활동 3회 이상 누적 시 활동 확인 배지 부여 |
| 인증 표시 | `○○구 거주 선언`, `○○구 활동 확인` 배지 |
| 제외 방식 | 공과금, 계약서 등 문서 인증 제외 |

---

### 12-3. 방문 인증

MVP에서는 **GPS 기반 간이 방문 인증**을 우선 구현한다.

다만 서버 배포 및 HTTPS 환경이 확보되지 않은 경우를 고려하여,  
댓글 기반 방문 인증을 대체 수단으로 제공한다.

#### GPS 기반 간이 방문 인증

```text
1. 사용자가 방문 인증 버튼 클릭
2. 프론트에서 브라우저 Geolocation API로 현재 위도/경도 획득
3. 프론트가 백엔드로 좌표 전송
4. 백엔드가 지역 중심 좌표와 사용자 좌표 간 거리 계산
5. 기준 반경 이내면 방문 인증 성공
```

MVP에서는 행정구역 경계 판별은 제외하고,  
지역 중심 좌표 기준 반경 계산으로 처리한다.

```text
기본 반경: 2km ~ 3km
```

#### 댓글 기반 대체 인증

GPS 사용이 불가능한 경우, 사용자는 해당 지역 Battle에 방문 경험 댓글을 작성한 뒤 방문 인증을 신청할 수 있다.

---

### 12-4. 방문 인증 구현 제약

브라우저 위치 정보는 보안 컨텍스트에서 동작한다.

따라서 MVP에서는 다음 기준으로 구현한다.

- 로컬 시연: `localhost` 환경에서 GPS 인증 가능
- 배포 시연: 프론트/백엔드 HTTPS 환경이 확보된 경우 GPS 인증 가능
- HTTPS 환경 미확보 시: 댓글 기반 방문 인증으로 대체

---

### 12-5. Reputation 점수 반영

| 인증 상태 | 해당 지역 댓글 가중치 | Insight 표시 레이어 |
|---|---:|---|
| 인증 없음 | 1.0 | 전체 사용자 결과 |
| 거주 선언만 완료 | 1.0 | 거주 선언 레이어 |
| 거주 선언 + 활동 확인 | 1.3 | 활동 확인 레이어 |
| 방문 인증 완료 | 1.2 | 방문자 레이어 |
| 거주 선언 + 방문 인증 | 1.3 | 활동 확인 + 방문자 레이어 |

가중치는 Insight 교차분석 데이터 필터링 기준에만 적용한다.  
Point 획득 배율에는 영향을 주지 않는다.

## 13. Insight / AI

Insight는 Battle, Market, Reputation에서 발생한 데이터를 분석하는 도메인이다.

### 13-1. 주요 역할

- Battle 결과 분석
- 사용자 그룹별 선호 분석
- 지역별 평판 데이터 생성
- Market 데이터 요약
- AI 분석 요청 관리
- 공공데이터 조회
- AI 분석 결과 저장

---

### 13-2. AI 데이터 흐름

MVP에서는 MCP를 직접 구현하지 않는다.

Insight-Reputation Service가 필요한 데이터를 직접 조회한 뒤,  
해당 데이터를 프롬프트에 포함하여 Claude API를 호출한다.

```text
Battle / Market
  → Insight-Reputation 분석 트리거
    → Battle / Market 원본 데이터 조회
    → 공공 API 데이터 조회
    → Claude API 호출
    → 분석 결과 저장
    → Client에 분석 리포트 제공
```

---

### 13-3. AI Battle 결과 요약

투표 종료 후 결과를 자연어로 요약한다.

예시:

```text
전체 투표에서는 성수가 61%로 우세했습니다.
다만 방문 인증 사용자 그룹에서는 연남 선호가 더 높게 나타났습니다.
댓글에서는 성수의 상권과 접근성이 장점으로 언급되었고,
연남은 분위기와 산책 코스가 강점으로 언급되었습니다.
```

---

### 13-4. AI Market 정보 요약

Market에서는 특정 선택지를 추천하지 않는다.

대신 사용자가 판단할 수 있도록 다음 정보를 요약한다.

- 관련 과거 통계
- 최근 추세
- YES 근거
- NO 근거
- 데이터 출처
- 판정 기준

예시:

```text
YES 근거:
최근 3주간 서울 아파트 매매가격지수는 상승세를 보였습니다.

NO 근거:
거래량은 여전히 낮은 수준이며, 일부 지역은 보합세를 유지하고 있습니다.

주의:
AI는 예측 방향을 추천하지 않으며, 사용자의 판단을 돕기 위한 정보만 제공합니다.
```

## 14. 공공 API 장애 및 예외 처리

### 14-1. 공공 API 장애 시 Market 처리

| 상황 | 처리 |
|---|---|
| 정산일 당일 API 데이터 미수신 | 자동 재시도 또는 관리자 확인 전환 |
| 재시도 실패 | Market 상태를 `DATA_PENDING`으로 변경 |
| 정산일 이후 최대 7일 내 데이터 공개 | 정상 정산 |
| 7일 이후에도 데이터 미공개 | Market 무효 처리 |
| 부분 데이터 누락 | 판정 불가 시 무효 처리 |
| 데이터 값 이상 | 관리자 검토 후 정산 또는 무효 처리 |

---

### 14-2. 무효 처리 흐름

```text
관리자 → Market                  PATCH /markets/{marketId}/void
Market → Member-Point            POST /points/refunds
Member-Point → DB                참여자별 참여 Point 전액 환불
Market → DB                      Market 상태 VOIDED 전환
Market → Client                  무효 처리 결과 표시
```

---

### 14-3. 관리자 정산 오류 처리

| 발견 시점 | 처리 |
|---|---|
| 정산 실행 전 | 판정 결과 수정 후 정산 실행 |
| 정산 실행 후 24시간 이내 | 정산 취소 → 지급 Point 회수 → 재정산 |
| 정산 실행 후 24시간 초과 | 결과 변경 불가, 필요 시 Point 차액 수동 보정 |

## 20. 기대 효과

동네대전은 단순히 지역 정보를 보여주는 서비스가 아니다.

사용자의 선택, 경험, 예측을 데이터로 축적하여 다음과 같은 가치를 제공한다.

1. 지역 선택에 대한 집단지성 제공
2. 연령대/성별/거주지별 선호 차이 시각화
3. 방문 인증 기반 신뢰도 있는 의견 제공
4. 공공데이터 기반 지역 예측 시장 제공
5. AI를 통한 투표 결과와 시장 정보 요약
6. Point 기반 참여 경제를 통한 사용자 리텐션 강화
7. 커뮤니티 활동을 통한 지역 평판 데이터 축적

최종적으로 동네대전은  
**“어디가 더 좋은가?”라는 막연한 질문을 데이터 기반 의사결정으로 바꾸는 플랫폼**을 목표로 한다.
